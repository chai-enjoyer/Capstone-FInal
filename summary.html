1. Software Processes & Models

  • Software Process — A structured set of activities required to develop a software system.
  • Four Universal Activities — Specification, Design & Implementation, Validation, Evolution.
  • Process Models — Abstract representations describing a process from a specific perspective.

Process Description Components

ComponentDescription
ProductsTangible outcomes — architectures, interface specs, data models
RolesResponsibilities assigned — Architect, Developer, Project Manager
Pre/Post-conditionsLogical statements that must hold before/after an activity

Plan-Driven vs. Agile

ApproachPlanningProgress Measurement
Plan-DrivenAll activities planned in advanceTrack against the plan
AgileIncremental, evolving planningFlexibility over roadmap

Most practical processes are hybrids — the goal is finding the right balance of discipline and agility.

2. The Waterfall Model

The archetype of plan-driven development. Standard for large, multi-site systems engineering projects.

  1. Requirements Analysis & Definition — Freeze system services, constraints, goals
  2. System & Software Design — Partition requirements, establish architecture
  3. Implementation & Unit Testing — Realize design as program units
  4. Integration & System Testing — Combine units, test for emergent properties
  5. Operation & Maintenance — Deploy and evolve as new requirements emerge

Critical flaw: Inflexible Partitioning — struggles to accommodate change. Only appropriate when requirements are stable and well-understood.

3. Incremental Development

Interleaves specification, development, and validation. System produced as a series of versions.

BenefitBusiness Outcome
Reduced Cost of ChangeLess rework compared to Waterfall
Enhanced Customer FeedbackStakeholders comment on working demos
Rapid DeliveryCustomers gain value from early increments

Pain Points

  • Process Visibility — Managers struggle because constant documentation isn't cost-effective
  • Structure Degradation — Frequent changes corrupt architecture; requires refactoring

4. Integration & Configuration (Reuse)

Modern systems are assembled from existing, configurable elements rather than built from scratch.

Types of Reusable Software

COTS (Commercial off-the-shelf) Component Frameworks (.NET, J2EE) Web Services (remote invocation)

Trade-offs

  • Benefit: Reduces cost and risk
  • Cost: Requirement Compromises — system may not meet 100% of user needs
  • Cost: Loss of Control — external vendors manage updates to reused elements

5. Requirements Engineering

Produces the Requirements Document through three sub-activities:

  1. Elicitation & Analysis — What stakeholders expect
  2. Specification — Defining requirements in detail
  3. Validation — Checking if requirements are realistic

Design & Implementation Model

ActivityOutput
Architectural DesignSystem Architecture
Database DesignDatabase Specification
Interface DesignInterface Specification
Component DesignComponent Specification

Validation & Verification (V&V)

  • Component Testing — Individual functions/objects tested independently
  • System Testing — Whole system; finds emergent properties
  • Customer Testing — Validating with real customer data
  • Debugging — Finding/correcting faults (distinct from V&V)

7. Structural & Behavioral Models

Class Diagrams

Define system objects with attributes, operations, and associations.

Consultation Class Example:

  • Attributes: Doctors, Date, Time, Clinic, Reason, Medication, Treatment, Voice notes, Transcript
  • Operations: New(), Prescribe(), RecordNotes(), Transcribe()

Complexity Management

TechniquePurposeBenefit
Generalization (Inheritance)Superclasses hold common features; subclasses add specific onesChanges only need to examine the superclass
Aggregation"Part-of" relationships; compositions of classesShows structural dependencies

Behavioral Models: Stimuli Types

TypeCharacteristicsExample
Data-DrivenControlled by data input, minimal external eventsEnd-to-end processing pipelines
Event-DrivenFinite states, stimuli trigger transitionsMicrowave oven, phone switching

Microwave Oven State Machine

  • States: Waiting → Half Power → Full Power → Set Time → Enabled → Operation → Disabled
  • Superstates group related states for complexity management
  • Stimuli trigger transitions between states (e.g., "full power" button, timer, door open)

8. Model-Driven Engineering (MDE/MDA)

Software development approach where models (not programs) are the primary output.

Three Model Types (MDA)

ModelFull NamePurpose
CIMComputation Independent ModelDomain concepts, no computation details
PIMPlatform Independent ModelSystem operations without platform specifics
PSMPlatform Specific ModelTailored to specific implementation platform

Key Concepts

  • Model Transformations: CIM → PIM → PSM → Code (automated translations)
  • Executable UML: UML models can be compiled directly into executable code
  • Promise: Raise abstraction level, reduce implementation errors
  • Limitation: Complete automation is still challenging for complex systems

9. Safety Engineering

Safety — ability to operate without causing human injury/death or environmental damage. A system property, not a component feature.

The Reliability vs. Safety Paradox

A system can be 100% reliable yet fundamentally unsafe. Three causes:

  1. Specification Errors — design itself contains a flaw
  2. Hardware Failures — spurious inputs overwhelm the specification
  3. Context-Sensitive Commands — correct command at wrong time

Safety-Critical System Classification

TypeDescriptionExample
PrimaryFailure directly threatens human lifeInsulin Pump Control System
SecondaryFailure causes faults in other systemsMentcare System, Grid Control

Safety Terminology

TermDefinitionInsulin Pump Example
Accident (Mishap)Unplanned event → death/injury/damageInsulin overdose
HazardCondition with potential for accidentBlood glucose sensor failure
DamageLoss measure from a mishapSerious injury or death
RiskProbability × severityLow to Medium

Risk Triangle

Intolerable — must be eliminated ALARP — As Low As Reasonably Practical Acceptable — negligible consequences

Hazard-Driven Strategies

StrategyAction
Eradicate (Avoidance)Design so hazards cannot arise
Intercept (Detection)Identify hazards in real-time, mitigate before accident
Insulate (Damage Limitation)Minimize fallout (relief valves, fail-safe modes)

Insulin Pump Safety Requirements (SR1–SR6)

  1. SR1: Shall not deliver a single dose > specified maximum
  2. SR2: Shall not deliver daily cumulative dose > maximum
  3. SR3: Hardware diagnostics ≥ 4 times per hour
  4. SR4: Exception handlers for all identified exceptions
  5. SR5: Audible alarm on any hardware/software anomaly
  6. SR6: Delivery suspended in alarm state until manual reset

Agile Liability for Safety-Critical Systems

  • Regulatory Compliance: Regulations demand extensive documentation; agile's "software over documentation" creates liability
  • Architectural Debt: Agile's interleaved development prevents holistic hazard analysis

10. Dependability, Reliability & V&V

Fault → Error → Failure Chain

  • Fault: Root cause (defect in code/spec)
  • Error: Incorrect internal state caused by a fault
  • Failure: Observable deviation from expected behavior

Advanced V&V Techniques

TechniquePurposeLimitation
Formal MethodsMathematical specification; finds spec errors, race conditionsExtremely expensive; proofs can contain errors
Model CheckingExplores all paths through a model; verifies propertiesState space explosion for large systems
Static AnalysisParses code without execution; finds error patternsFalse positives; cannot find all runtime errors

Static Analysis Fault Categories

CategoryExamples
Data FaultsVariables used before initialization; declared but never used
Control FaultsUnreachable code; unconditional branches into loops
I/O FaultsVariables output twice with no intervening assignment
Interface FaultsParameter type/number mismatches; uncalled procedures
Storage Mgmt FaultsUnassigned pointers; pointer arithmetic errors
